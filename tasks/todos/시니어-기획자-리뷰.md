## 요약 (Executive Summary)

Align Agents v2는 여러 AI 코딩 어시스턴트를 사용하는 개발자들의 **실제 문제를 해결하는 잘 설계된 데스크톱 애플리케이션**입니다. 기술적 기반이 탄탄하고 UX 결정이 신중합니다. 그러나 런칭 전 반드시 해결해야 할 핵심 사용성 이슈가 존재합니다.

## Part 1: 제품 비전 및 전략 분석

### 1.1 문제 정의 (우수)

이 제품은 점점 커지는 개발자들의 고충을 정확히 파악하고 있습니다:
- 개발자들이 이제 3~5개 이상의 AI 코딩 어시스턴트 사용 (Cursor, Windsurf, Claude, Cline 등)
- 각 도구마다 설정 형식과 저장 위치가 다름
- 여러 도구에서 규칙과 MCP 서버를 일관되게 유지하는 것이 번거로움
- 통합 관리 솔루션이 기존에 없음

**강점**: 문제가 실제로 존재하고, 계속 커지고 있으며, 아직 해결되지 않았습니다.

### 1.2 솔루션 접근법 (양호, 단서 있음)

"통합 설정 허브" 접근 방식은 합리적입니다:

```
+----------------+     +-----------------+     +----------------+
| 규칙 & MCP     | --> | Set으로 구성     | --> | 도구에 동기화   |
| 서버 생성      |     | (Rules/MCP)     |     | (Cursor 등)    |
+----------------+     +-----------------+     +----------------+
```

**강점**:
- 중앙 집중식 관리로 중복 제거
- Set 기반 그룹핑으로 유연한 배포 가능
- 동기화 전 미리보기로 실수 방지
- History/롤백으로 안전망 제공

**우려사항**:
1. **범위 복잡성**: 3단계 범위(Global/Project/Project-Local)가 사용자를 혼란스럽게 할 수 있음
2. **멘탈 모델 차이**: 사용자는 "이 규칙을 Cursor에 동기화하고 싶어"라고 생각하지만, 제품은 "Set 생성 → 도구에 할당 → 동기화" 흐름을 요구

### 1.3 타겟 사용자 분석

**주요 페르소나** (잘 정의됨):
- 2개 이상의 AI 어시스턴트를 사용하는 파워 개발자
- 터미널/설정에 익숙함
- 일관성과 자동화를 중시

**누락된 페르소나**:
- **캐주얼 사용자**: 그냥 "내 모든 도구가 똑같이 작동하게 해줘"를 원하는 개발자
- **팀 리더**: 팀원들에게 표준화된 설정을 배포하고 싶은 PM/팀장
- **AI 도구 입문자**: 첫 AI 어시스턴트를 사용해보는 사람, 가이드 필요

**권고**: Set 관리가 필요 없는 캐주얼 사용자를 위한 "빠른 동기화" 모드 추가 고려.

## Part 2: 정보 구조 리뷰

### 2.1 네비게이션 구조 (7개 메뉴)

```
Platform 섹션:
├── Tools      ← 시작점? 아니면 Sync? 불명확
├── Sync       ← 핵심 액션, 하지만 사전 설정 필요
├── Rule Sets  ← 콘텐츠 생성
└── MCP Sets   ← 콘텐츠 생성

Workspace 섹션:
├── Projects   ← 프로젝트 레벨 동기화에 필요
└── History    ← 참조/복구

Settings       ← 표준 위치
```

**발견된 이슈**:

1. **불명확한 진입점**: 신규 사용자가 어디서 시작해야 할지 모름
   - Tools 페이지는 감지된 도구를 보여주지만 다음 단계를 안내하지 않음
   - Sync 페이지는 규칙/MCP Set이 먼저 존재해야 함
   - 온보딩 마법사나 빈 상태 가이드 없음

2. **동사 vs 명사 혼란**:
   - "Sync"는 액션이고 나머지는 명사
   - 고려: "Sync Center" 또는 "배포"로 변경

3. **숨겨진 계층 관계**: Rules와 MCP가 별도 메뉴지만 Sync에서 함께 사용됨
   - 사용자가 이 개념들을 머릿속에서 연결해야 함
   - 관계를 시각적으로 표시하지 않음

### 2.2 개념적 계층 (3단계 규칙)

```
Level 1: Piece (원자적 규칙)
    ↓
Level 2: Composite (규칙 그룹)
    ↓
Level 3: Set (배포 가능한 컬렉션)
```

**분석**:

이 3단계 계층은 **기술적으로 우아하지만 인지적으로 부담**이 됩니다:

| 레벨 | 사용자 인식 | 실제 |
|------|------------|------|
| Piece | "하나의 규칙" | 가장 작은 단위 |
| Composite | "규칙 그룹" | 순서가 있는 하위 컬렉션 |
| Set | "배포하는 것" | 최종 패키지 |

**문제점**: 대부분의 사용자는 Piece + Set만 사용할 것입니다. Composite는 일반적인 워크플로우에서 명확한 이점 없이 복잡성만 추가합니다.

**권고**:
- v1에서는 2단계(Rule + Set)로 단순화 고려
- 또는 Composite를 "고급" 토글 뒤에 숨기기

### 2.3 범위 모델 (Global/Project/Project-Local)

```
Scope: Global          → ~/.cursor/mcp.json (모든 프로젝트)
Scope: Project         → .cursor/rules/*.mdc (git 추적)
Scope: Project-Local   → .cursor/rules/.local/ (gitignore)
```

**사용자 여정의 공백**:

사용자가 규칙을 동기화하려 할 때 직면하는 선택:
1. 어떤 범위? (3가지 선택)
2. 어떤 도구? (N가지 선택)
3. 어떤 Rule Set? (M가지 선택)
4. 어떤 MCP Set? (K가지 선택)

이는 **N × M × K × 3**개의 잠재적 설정 조합을 만듭니다. 사용자는 옵션이 아니라 가이드가 필요합니다.

**권고**: "Project" 범위를 기본값으로 하고 확장 옵션 제공.

## Part 4: 사용성 평가

### 4.1 태스크 분석: 신규 사용자 첫 설정

**목표**: 사용자가 자신의 코딩 규칙을 Cursor에 동기화하고 싶음

```
최적 경로 (전문가):
1. Rules 페이지 → 규칙 생성 → 저장
2. Rules 페이지 → Set 생성 → 규칙 추가 → 저장
3. Sync 페이지 → 도구 선택 → Set 선택 → 미리보기 → 확인

실제 경로 (초보자):
1. 앱 열기 → Tools 페이지 표시 → "이제 뭐하지?"
2. 이해하려고 여러 메뉴 무작위 클릭
3. 결국 Rules 찾음 → 빈 화면 → "어디서 시작하지?"
4. 규칙 생성 → Set 미생성 → Sync로 이동
5. Sync 페이지에 옵션 없음 → 혼란
6. Rules로 돌아가기 → Set 이해
7. Sync로 다시 → 성공 (아마도)

첫 동기화까지 시간:
- 전문가: ~3분
- 초보자: 15-30분 (포기하지 않으면)
```

**핵심 공백**: 안내된 첫 실행 경험 없음.

   - 영향: 규칙/Set 없이 Sync 진입
   - 권고: 요구사항 표시 또는 생성 안내

## Part 5: 사용자 경험 평가

### 5.1 시각 디자인 (우수)

"Linear 스타일" 디자인 시스템이 잘 실행되었습니다:
- 다크 모드 우선은 개발자 선호와 일치
- 미묘한 그라디언트와 글로우가 프리미엄 느낌 생성
- 일관된 간격과 타이포그래피
- 어수선함 없이 높은 정보 밀도

**한 가지 우려**: 일부 상호작용 요소(특히 드래그 핸들)가 너무 미묘함.

### 5.2 인터랙션 디자인 (양호)

| 패턴 | 평가 |
|------|------|
| 롱 프레스로 드래그 | 깔끔한 디자인에 영리함, 발견 가능성? |
| 크기 조절 가능한 패널 | 파워 유저 기능, 잘 작동 |
| 동기화 전 미리보기 | 신뢰 구축, 우수 |
| 토스트 알림 | 적절함, 방해되지 않음 |

### 5.3 감정적 여정

```
신규 사용자 여정:
┌─────────────────────────────────────────────────────────────┐
│ 기대감 → 혼란 → 좌절 → (포기 또는 학습)                      │
│    ↓                              ↓        ↓               │
│ "전문가처럼                    "그만둠"  "드디어             │

└─────────────────────────────────────────────────────────────┘

바람직한 여정:
┌─────────────────────────────────────────────────────────────┐
│ 기대감 → 안내된 설정 → 첫 성공 → 숙달                        │
│    ↓          ↓          ↓         ↓                       │
│ "전문가    "쉽네"     "작동해!"  "정말 효율적"               │

└─────────────────────────────────────────────────────────────┘
```

### 5.4 경쟁 포지셔닝

| 경쟁자 | 접근법 | Align Agents 장점 |
|--------|--------|------------------|
| 수동 설정 | 파일 직접 편집 | 자동화, 일관성 |
| IDE 설정 동기화 | 내장, 단일 도구 | 크로스 도구, AI 특화 |
| dotfiles 관리자 | 범용, 기술적 | AI 도구 맞춤형 |

**고유 가치**: AI 어시스턴트 설정 관리에 특화된 유일한 솔루션.

## Part 6: 권고사항

### 즉시 (베타 전)

1. **첫 실행 온보딩 추가**
   ```
   Step 1: "AI 도구를 감지해봅시다" [스캔]
   Step 2: "첫 번째 규칙을 만들어봅시다" [간단 입력]
   Step 3: "도구에 동기화하세요" [원클릭 배포]
   ```

2. **빈 상태 개선**
   - 모든 빈 리스트에: 설명 + CTA + 예시

3. **Sync 페이지 안내 추가**
   - Set이 없으면: "먼저 규칙이나 MCP Set을 만들어야 합니다. [Rules로 이동]"
   - Sync가 무엇을 하는지 설명하는 인라인 도움말

### 단기 (GA 전)

4. **캐주얼 사용자를 위한 단순화**
   - "빠른 동기화" 모드: Set 건너뛰고 규칙 직접 동기화
   - 고급 기능(Composite 규칙, 3개 범위) 토글 뒤에 숨기기

5. **인앱 도움말 추가**
   - 모든 주요 UI 요소에 호버 시 툴팁
   - 컨텍스트 도움말 패널을 여는 "?" 버튼
   - 빈 상태에서 비디오 튜토리얼 링크

6. **MCP 서버 템플릿**
   - 일반적인 MCP 서버를 위한 사전 구성된 템플릿
   - 유용한 오류 메시지와 검증

### 장기 (GA 이후)

7. **팀 기능**
   - URL/QR 코드로 Rule Set 공유
   - 조직 수준 설정
   - 역할 기반 접근 (관리자가 특정 규칙 잠금 가능)

8. **지능형 기능**
   - 프로젝트 타입에 따른 규칙 제안
   - 도구 간 충돌 감지
   - 기존 설정에서 자동 가져오기

9. **분석 대시보드**
   - 가장 많이 사용되는 규칙 추적
   - 동기화 성공/실패 모니터링
   - 설정 드리프트 식별

## Part 7: 결론

### 유지해야 할 강점

1. **명확한 제품 비전**: 실제로 증가하는 문제 해결
2. **프리미엄 디자인 품질**: Linear 스타일 실행 우수
3. **견고한 아키텍처**: 깔끔한 코드로 향후 반복 가능
4. **안전 기능**: 미리보기 + History로 사용자 신뢰 구축
5. **확장성**: 플러그인 패턴으로 새 도구 추가 가능

### 해결해야 할 핵심 공백

1. **온보딩**: 신규 사용자에게 안내 없음
2. **발견 가능성**: 기능은 있지만 명확하지 않음
3. **인지 부하**: 처음에 배워야 할 개념이 너무 많음
4. **문서화**: 인앱 도움말 거의 없음

### 최종 판정

**Align Agents v2는 집중적인 UX 개선으로 9/10이 될 수 있는 7/10 제품입니다.**

기술적 기반이 견고하고, 디자인이 프리미엄이며, 해결하는 문제가 실재합니다. 그러나 현재 UX는 신규 사용자에게 불필요한 마찰을 만듭니다. 권고된 온보딩과 안내 개선으로, 이 제품은 개발자 커뮤니티에서 채택될 강력한 잠재력을 가지고 있습니다.

**권고**: P0 사용성 수정을 구현하기 위해 퍼블릭 베타를 2-4주 연기하세요. 지금 UX 이슈를 수정하는 비용은 나쁜 첫인상에서 회복하려는 것보다 훨씬 낮습니다.
